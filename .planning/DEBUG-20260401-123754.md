## PROBLEMA 1 — Bug "todo imóvel aparece como APTO 1"

```
SINTOMA: Ao abrir o form de edição de qualquer imóvel, o nome exibido
         é sempre "Apto 01" — independente do imóvel real.

HIPÓTESE: gerarNome() usa pavimento/numeroUnidade do estado local
          (inicializado com defaults 1/1), não os valores do banco.

TESTE: Leitura do useEffect em imovel-form-panel.tsx linha 121.
```

**CAUSA RAIZ CONFIRMADA — linha 121-122 de `imovel-form-panel.tsx`:**

```tsx
// useEffect que popula o form na edição:
if (open && imovel) {
  setPavimento(imovel.pavimento ?? 1)         // linha 122 — correto
  setNumeroUnidade(imovel.numero_unidade ?? 1) // linha 123 — correto
  ...
}
```

O useEffect está CORRETO — ele lê `imovel.pavimento` e `imovel.numero_unidade` do banco.

**O problema real é anterior: os dados no banco estão com `pavimento = null` ou `= 1` e `numero_unidade = null` ou `= 1` para TODOS os imóveis existentes.**

Evidência — `001_schema.sql.sql` linha 99-109: a tabela `imoveis` foi criada originalmente SEM as colunas `pavimento`, `cobertura` e `numero_unidade`. Elas só foram adicionadas pela migration `003_imoveis_pavimento.sql`, e o SEED de 6 imóveis (linhas 335-343) foi inserido ANTES disso, sem esses valores.

Resultado: os 6 imóveis do seed têm `pavimento = 1` (default) e `numero_unidade = 1` (default) em todos os registros, então o `useEffect` lê `1` e `1` corretamente — mas o dado do banco é incorreto.

Quando o `useEffect` faz `setPavimento(imovel.pavimento ?? 1)`:
- `imovel.pavimento` vale `1` para todos (default da migration 003)
- `imovel.numero_unidade` vale `1` para todos (default da migration 003)
- `gerarNome(1, 1)` retorna `"Apto 01"` para todos

**Conclusão:** o bug não é de código — é de dado. Todos os 6 imóveis do seed têm `pavimento=1` e `numero_unidade=1` porque foram inseridos antes das novas colunas existirem e o SQL de seed não foi re-executado com os valores corretos.

**Fix necessário (SQL):**
```sql
-- Atualizar os imóveis existentes com os valores corretos de pavimento/unidade
-- Os IDs originais eram text ('Apto_01', etc.) mas a API nova usa UUID —
-- verificar qual é o id real antes de rodar.

-- Se os imóveis ainda têm os IDs originais do seed:
UPDATE imoveis SET pavimento = 1, numero_unidade = 1, cobertura = false WHERE id = 'Apto_01';
UPDATE imoveis SET pavimento = 2, numero_unidade = 1, cobertura = false WHERE id = 'Apto_02';
UPDATE imoveis SET pavimento = 3, numero_unidade = 1, cobertura = false WHERE id = 'Apto_03';
UPDATE imoveis SET pavimento = 4, numero_unidade = 1, cobertura = false WHERE id = 'Apto_04';
UPDATE imoveis SET pavimento = 9, numero_unidade = 1, cobertura = true  WHERE id = 'Cob_01';
UPDATE imoveis SET pavimento = 9, numero_unidade = 2, cobertura = true  WHERE id = 'Cob_02';
```

**Problema adicional no seed — IDs incompatíveis com a API nova:**

O seed original (`001_schema.sql.sql` linha 100) define `id text PRIMARY KEY` com valores como `'Apto_01'`. A API `POST /api/imoveis` (`route.ts` linha 45) gera `id: crypto.randomUUID()`. A migration `003` não alterou o tipo do campo `id`. Isso significa:

- Se o banco ainda tem a tabela original (id TEXT), o `crypto.randomUUID()` vai inserir um UUID em campo TEXT — vai funcionar, mas é inconsistente.
- Se a tabela foi recriada sem o seed antigo, está vazia.

**Ação:** verificar `SELECT id, pavimento, numero_unidade FROM imoveis;` no Supabase para confirmar o estado real.

## PROBLEMA 2 — Erros ao salvar imóveis (POST/PUT)

```
SINTOMA: Salvar imóvel retorna erro.

HIPÓTESE 1: Tabela imoveis não tem as colunas pavimento/cobertura/numero_unidade
            (migration 003 não foi aplicada).

HIPÓTESE 2: RLS bloqueia o INSERT/UPDATE para adm.
```

**TESTE 1 — Verificar migration 003:**

A migration `003_imoveis_pavimento.sql` usa `ADD COLUMN IF NOT EXISTS`, portanto é segura de re-executar. Se ela não foi aplicada, qualquer INSERT que inclua `pavimento`, `cobertura` ou `numero_unidade` vai retornar:

```
column "pavimento" of relation "imoveis" does not exist
```

A API (`route.ts` linha 54-56) sempre envia essas três colunas:
```ts
pavimento: body.pavimento ?? 1,
numero_unidade: body.numero_unidade ?? 1,
cobertura: body.cobertura ?? false,
```

Se a migration 003 não foi aplicada, **todo POST e PUT de imóvel falha com 500**.

**TESTE 2 — Verificar RLS:**

A policy `"imoveis: apenas ADM gerencia"` (001_schema.sql.sql linha 326-330):
```sql
CREATE POLICY "imoveis: apenas ADM gerencia"
  ON imoveis FOR ALL
  TO authenticated
  USING (is_adm())
  WITH CHECK (is_adm());
```

A função `is_adm()` usa `SECURITY DEFINER` e consulta `user_profiles`. Se o usuário logado não tem registro em `user_profiles` com `role = 'adm'`, a policy bloqueia.

A API verifica role via código (`requireAdm`) antes de chamar o Supabase, mas usa `createClient()` que conecta com a **anon key** (não service role). Portanto o RLS ainda é aplicado pelo Supabase mesmo que o código já passou pela verificação.

**CAUSA RAIZ PROVAVEL:** a RLS policy usa `FOR ALL` que cobre SELECT, INSERT, UPDATE e DELETE com `USING (is_adm())`. Uma policy com `USING` aplica-se ao SELECT e como filtro de linha no UPDATE/DELETE. Para INSERT, apenas `WITH CHECK` é avaliado — o `USING` é ignorado. Portanto o INSERT está coberto. O UPDATE também tem `WITH CHECK (is_adm())`. Isso deve funcionar desde que o usuário seja adm.

**CAUSA RAIZ MAIS PROVAVEL:** migration `003_imoveis_pavimento.sql` não foi aplicada ao banco.

**Fix necessário (SQL):**
```sql
-- Executar no Supabase SQL Editor:
ALTER TABLE imoveis
  ADD COLUMN IF NOT EXISTS cobertura BOOLEAN NOT NULL DEFAULT false,
  ADD COLUMN IF NOT EXISTS numero_unidade SMALLINT NOT NULL DEFAULT 1;

DO $$
BEGIN
  IF NOT EXISTS (
    SELECT 1 FROM information_schema.columns
    WHERE table_name = 'imoveis' AND column_name = 'pavimento'
  ) THEN
    ALTER TABLE imoveis ADD COLUMN pavimento SMALLINT NOT NULL DEFAULT 1;
  END IF;
END $$;

ALTER TABLE imoveis
  DROP CONSTRAINT IF EXISTS imoveis_pavimento_check,
  DROP CONSTRAINT IF EXISTS imoveis_numero_unidade_check;

ALTER TABLE imoveis
  ADD CONSTRAINT imoveis_pavimento_check CHECK (pavimento BETWEEN 1 AND 9),
  ADD CONSTRAINT imoveis_numero_unidade_check CHECK (numero_unidade BETWEEN 1 AND 4);
```

## PROBLEMA 3 — Erros ao salvar etiquetas (labels)

```
SINTOMA: Salvar etiqueta retorna erro.

HIPÓTESE 1: Tabela labels não existe (migration 002 não foi aplicada).
HIPÓTESE 2: RLS bloqueia INSERT porque created_by != auth.uid().
```

**CAUSA RAIZ 1 — migration 002 pode não existir no banco:**

Se `002_labels.sql` não foi executada, qualquer request para `GET /api/labels` retorna:
```
relation "labels" does not exist
```

**CAUSA RAIZ 2 (confirmada por código) — bug na policy INSERT de labels:**

`002_labels.sql` linha 23-24:
```sql
CREATE POLICY "labels_insert" ON labels FOR INSERT TO authenticated
  WITH CHECK (auth.uid() = created_by);
```

A API `POST /api/labels` (`src/app/api/labels/route.ts` linha 29-31):
```ts
const { data, error } = await supabase
  .from('labels')
  .insert({ name: body.name.trim(), color: body.color ?? '#6366F1' } as never)
```

**O INSERT não envia `created_by`**. A policy exige `auth.uid() = created_by`, mas como `created_by` não é enviado, o Supabase recebe `NULL` para esse campo. A comparação `auth.uid() = NULL` é sempre `false` em SQL. O RLS bloqueia o INSERT com `new row violates row-level security policy`.

Este é o erro que o usuário está vendo ao tentar criar etiquetas.

**Fix necessário (código):**

Em `src/app/api/labels/route.ts` linha 29, adicionar `created_by: user.id` ao insert:
```ts
.insert({ name: body.name.trim(), color: body.color ?? '#6366F1', created_by: user.id } as never)
```

**OU** alterar a policy para não exigir created_by no INSERT:
```sql
DROP POLICY IF EXISTS "labels_insert" ON labels;
CREATE POLICY "labels_insert" ON labels FOR INSERT TO authenticated
  WITH CHECK (true);
```

A opção de código é mais correta pois mantém a rastreabilidade de quem criou cada etiqueta.

## PROBLEMA 4 — Erros ao salvar leads (POST /api/leads)

```
SINTOMA: Salvar lead manual retorna erro.

HIPÓTESE: O campo phone tem UNIQUE constraint. Tentativa de criar lead
          com telefone já existente retorna 409/500.
```

**Verificação — `001_schema.sql.sql` linha 31:**
```sql
phone text NOT NULL UNIQUE,
```

**`src/app/api/leads/route.ts` — não trata erro de unicidade.** Se o phone já existe, o Supabase retorna erro `23505 (unique_violation)` e a API retorna `{ error: "duplicate key value..." }` com status 500.

Este não é um bug de código, mas falta de tratamento amigável. Para o caso de um lead ser criado via webhook (n8n/WhatsApp) antes do corretor criar manualmente com o mesmo número, o erro aparece.

A API de leads não tem outros problemas estruturais aparentes — a tabela `leads` existia na migration original e não depende das migrations 002 ou 003.
